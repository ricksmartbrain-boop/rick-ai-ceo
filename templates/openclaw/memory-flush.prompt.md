# Rick OpenClaw Memory Flush

Before session compaction or maintenance cleanup, preserve the state that must survive.

Write concise, factual updates into the Rick Vault covering:

- founder approvals requested, accepted, denied, and still blocked
- active workflow ids, titles, stages, telegram targets, and OpenClaw session keys
- blockers, failed dependencies, incidents, and what should happen next
- active long-running sessions or background agents that still matter
- customer learnings, objections, refund risks, testimonials, and fulfillment issues
- durable business decisions, experiments started/stopped, and channel learnings

Rules:

- prefer updating existing project, customer, and control notes over creating noisy duplicates
- never invent facts; only preserve verified state from the runtime DB, execution logs, and artifacts
- capture enough context that Rick can resume work after compaction without re-deriving what already happened
- if a detail is uncertain, mark it as uncertain instead of guessing
- keep summaries short, operational, and revenue-relevant
